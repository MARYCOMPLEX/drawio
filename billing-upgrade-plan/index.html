MISSING 缺失能力 — 需要新建
1. 账户冻结余额
有: agents.balance, users.remained_quota
缺: frozen_balance / frozen_quota 字段
预扣直接减余额，没有冻结分离。无法区分"真的被扣了"和"只是冻住了"。
2. 统一账本 ledger_entry
有: agent_transactions, user_transactions
缺: 统一 ledger_entry 表，带 PREAUTH/SETTLE/REFUND 类型
现有 transaction 表是"余额快照"模式，不是"对称借贷"模式。无法追溯一笔请求的完整扣费链条。
3. request_settlement 结算单
有: logs 表里有各层金额
缺: 独立 request_settlement 表固化三方金额+利润
利润是事后从 log 聚合算出来的，不是交易时写入的。对账困难。
4. Outbox 事件表
有: 无
缺: event_outbox + Dispatcher
结算结果无法异步推送到下游。统计靠 cron 全表扫描 log，无事件驱动。
5. Kafka 事件总线
有: 无
缺: Kafka + billing.request.settled topic
交易和统计完全耦合在 PG。报表查询会影响主库。
6. ClickHouse 分析库
有: billing_daily_stats 在 PG
缺: ClickHouse 明细表 + 分钟聚合 + 日聚合
所有聚合都在 PG 里做。30 分钟 cron 延迟大，无实时看板能力。
7. 平台真实成本确认
有: L0 ratio 估算成本
缺: 实际上游计费确认 + COST 账本
成本基于 ratio 推算，不是上游真实账单。无法精确计算平台利润。
8. 对账校验公式
有: AgentProfit = L4 - L3 (在 log 里)
缺: 总毛利 = 代理商利润 + 平台利润 的自动校验
利润数字分散在 log 各字段，无系统级对账。

SCHEMA 新增 PG 表结构
表名核心字段说明
ledger_entry id, request_id, account_type(user/agent/platform), account_id, entry_type(PREAUTH/SETTLE/REFUND/EXTRA_CHARGE/COST), amount, balance_before, balance_after, frozen_before, frozen_after, model_name, created_at 每个账务动作一条。同一 request_id 可有多条（预扣+实扣+退差）。
request_settlement id, request_id, agent_id, user_id, model_name, channel_id, provider_id, actual_user_amt, actual_agent_amt, actual_platform_cost, agent_profit, platform_profit, prompt_tokens, completion_tokens, latency_ms, status, settled_at 一笔请求一条。财务真相。利润公式校验来源。
event_outbox id, event_type, aggregate_id, payload(jsonb), status(PENDING/SENT/FAILED), retry_count, created_at, sent_at 事务内写入，Dispatcher 异步推 Kafka。保证至少一次投递。

agg_minute minute, agent_id, model, request_count, success_count, sum_tokens, sum_user_amt, sum_agent_amt, sum_platform_cost, sum_agent_profit, sum_platform_profit Materialized View 自动聚合。实时看板用。

ROADMAP 分步落地路线
STEP 1
PG 交易链路改造
在现有 BillingSession 基础上补齐账务能力，不引入外部依赖。
  • users 表加 frozen_quota
  • agents 表加 frozen_balance
  • 新建 ledger_entry 表
  • 新建 request_settlement 表
  • 改 BillingSession.preConsume 走冻结
  • 改 BillingSession.Settle 写 ledger + settlement
  • 保留老 log 写入不变（兼容）
STEP 2
Outbox + Kafka
结算结果通过事件总线异步送出，交易与统计解耦。
  • 新建 event_outbox 表
  • Settle 事务里追加写 outbox
  • 实现 Outbox Dispatcher goroutine
  • 引入 Kafka 客户端
  • Topic: billing.request.settled
  • Dispatcher 推 Kafka 后标 SENT
STEP 3
ClickHouse 明细
消费 Kafka 事件写入 ClickHouse，建立分析数据基础。
  • 部署 ClickHouse
  • 建 fact_request_settlement
  • 实现 Analytics Consumer
  • Consumer 消费 Kafka 写 CH
  • 灌入历史数据（从 PG logs + settlement 同步）
STEP 4
聚合 + 报表切换
分钟/日聚合 + 报表查询从 PG 切到 ClickHouse。
  • 建 agg_minute Materialized View
  • 建 agg_daily Materialized View
  • 改 billing_report 接口查 CH
  • PG billing_daily_stats 做降级备份
  • 对账校验: 总毛利 = 代理商利润 + 平台利润

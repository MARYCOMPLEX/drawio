## 1. 权限系统全景（5 分钟理解）

### 1.1 一句话总结

本系统有 **五道锁**，从粗到细控制用户能做什么、能看什么：

```
用户请求
  │
  ├── 第 1 道锁：菜单权限 ──── 侧边栏能不能看到这个入口？
  │
  ├── 第 2 道锁：API 权限 ──── 这个接口能不能调？
  │
  ├── 第 3 道锁：数据范围 ──── 能看到哪些行？（全平台 / 本代理商 / 仅自己）
  │
  ├── 第 4 道锁：字段权限 ──── 能看到行里哪些列？（成本价对销售隐藏）
  │
  └── 第 5 道锁：UI 权限 ───── 页面上的按钮、筛选器是否显示？
```

### 1.2 五道锁速查表

| 锁 | 控什么 | 后端怎么拦 | 前端怎么藏 | 谁来配 |
|---|---|---|---|---|
| 菜单权限 | 侧边栏菜单项 | — | `hasMenuPermission('menu:xxx')` 自动过滤 | 角色抽屉 · 功能权限 Tab |
| API 权限 | HTTP 接口调用 | 路由声明 `middleware.RequirePerm(code)` | `hasPermission(code)` / `<PermissionGate>` | 角色抽屉 · 功能权限 Tab |
| 数据范围 | 表里的行 | 路由声明 `middleware.WithResource("xxx")` → 自动生效 | `getDataScope('xxx')` 仅提示 | 角色抽屉 · 数据范围 Tab |
| 字段权限 | 行里的列 | `WithResource` 下 `ScopedList`/`ScopedGet` 自动脱敏 | `useFieldPermColumns()` 自动隐藏列 | 角色抽屉 · 字段权限 Tab |
| UI 权限 | 按钮/筛选器 | — | `<UIGate feature="x" element="y">` | 角色抽屉 · 交互权限 Tab |

### 1.3 框架级 RBAC 核心原则

1. **后端是安全边界**。前端所有隐藏/禁用都只是体验优化。
2. **路由声明，自动生效**。在路由加一行 `WithResource("xxx")`，Controller 调 `ScopedList` 即可——行级过滤 + 列级脱敏 + 分页响应全自动。
3. **`platform:*` 是上帝模式**。持有它的角色（root）绕过所有检查。**千万不要**给非 root 角色。
4. **字段权限是白名单**。新资源注册后，非 root 角色如果没有显式勾选字段，所有字段默认返回零值。
5. **出错就降级**。数据范围查不到 → 降级为 `WHERE 1=0`；字段权限出错 → 全部 deny。
6. **业务不决定权限**。Controller 里禁止写 `if user.Role == "admin"`。

### 1.4 系统内置角色

| code | 名称 | 说明 | 默认数据范围 |
|---|---|---|---|
| `root` | 总平台超管 | 持有 `platform:*`，上帝模式 | `all` |
| `admin` | 代理商超管 | 管理自己代理商的一切 | `agent`（本代理商） |
| `operator` | 运营 | 渠道、成本、模型管理 | — |
| `sales` | 销售 | CRM + 客户管理 | `self_and_sub`（自己+下属客户） |
| `presales` | 售前 | 只读 | `agent` |
| `finance` | 财务 | 可见财务字段 | `agent` |
| `user` | 普通用户 | 最低权限 | `self` |

## 2. 核心概念详解

### 2.1 Module（模块）

**模块是权限树的第一层分组**，用来把功能分类显示。

```
系统内置模块（定义在 permission_seed.go 的 defaultModules）：
├── chat       → 聊天（操练场、视频生成等）
├── console    → 控制台（令牌、用量、日志等用户功能）
├── personal   → 个人中心（充值、个人设置）
├── sales      → 销售（CRM、客户管理）
├── admin      → 管理后台（用户管理、渠道管理等）
├── platform   → 平台管理（代理商、角色、财务等）
└── audit      → 财务审计
```

### 2.2 Feature（功能）

**功能是权限的核心单元**。每个功能定义了：

```
FeatureSeed = {
  FeatureKey:      "procurement",              // 唯一标识
  FeatureName:     "采购中心",                  // 显示名称
  Module:          "platform",                 // 归属哪个模块
  Namespace:       "platform",                 // 权限码前缀
  RoutePath:       "/console/procurement",     // 前端路由
  Icon:            "Truck",                    // 图标（Lucide 图标名）
  Actions:         ["menu", "view", "create"], // 操作列表 → 自动派生权限码
  ApiCodeOverride: {"read": "custom:code"},    // 可选：覆盖默认派生规则
  Resources:       ["procurement_orders"],     // 关联的数据资源（数据范围 + 字段权限）
  UIElements:      [{Key:"export", ...}],      // UI 元素
}
```

### 2.3 Permission（权限码）—— 从 Feature 自动派生

**你不需要手动创建权限码！** 系统根据 Feature 的 Actions 自动生成（由 `DerivePermCode()` 函数执行）：

| Action 写 | 生成的权限码 | 类型 | 说明 |
|---|---|---|---|
| `menu` | `menu:{featureKey}` | 菜单 | 控制侧边栏是否显示 |
| `read` | `{namespace}:{featureKey}:view` | API | `read` 自动映射为 `view` |
| `view` | `{namespace}:{featureKey}:view` | API | 直接使用 `view` |
| `create` | `{namespace}:{featureKey}:create` | API | |
| `update` | `{namespace}:{featureKey}:update` | API | |
| `delete` | `{namespace}:{featureKey}:delete` | API | |
| 任意自定义 | `{namespace}:{featureKey}:{action}` | API | 如 `approve`、`export` |

> **ApiCodeOverride**：如果默认派生规则不满足需求，可用 `ApiCodeOverride` 字段覆盖。例如 `{"read": "tenant:log:view"}` 会让 `read` 动作使用指定的权限码而非默认派生。

### 2.4 Resource（资源）

**资源是数据范围和字段权限的操作对象**，通常对应一张数据库表。通过路由的 `WithResource("xxx")` 绑定。

### 2.5 UIElement（UI 元素）

**UI 元素控制页面上的具体交互组件**（按钮、筛选器、Tab 等）。

## 3. 关键流程图解

### 3.1 新功能上线后的完整流程

```
              开发阶段                                    部署后
       ──────────────────                         ──────────────────

  开发者写代码                                    管理员在后台配置
      │                                               │
      ▼                                               ▼
  ┌──────────────────┐   部署    ┌────────────────────────────────┐
  │ permission_seed   │ ───────→ │ 启动时 EnsureDefaultFeatures() │
  │ 定义 Feature      │          │ 自动创建 Feature + 权限码       │
  │                  │          │ 但 不会 自动分配给任何角色       │
  │ WithResource     │          └────────────────────────────────┘
  │ 路由绑定资源      │                        │
  │                  │                        ▼
  │ ScopedList       │          ┌────────────────────────────────┐
  │ Controller 1行搞定│          │ 管理员打开「角色与权限」页面     │
  └──────────────────┘          │ 看到新功能，默认未勾选          │
                                │ 按需配置：功能/范围/字段/UI     │
                                └────────────────────────────────┘
```

### 3.2 请求时的权限检查流程（框架自动执行）

```
用户发起请求: GET /api/procurement/orders
      │
      ▼
  ① middleware.UserAuth()              ← 路由声明，自动执行
      │  从 JWT 提取 userId, agentId
      │
      ▼
  ② middleware.WithResource("xxx")     ← 路由声明，自动执行
      │  将 ResourceConfig 写入 context
      │
      ▼
  ③ middleware.RequirePerm(code)        ← 路由声明，自动执行
      │  Casbin 查询权限（30s 缓存） → 没有 → 403
      │
      ▼
  ④ Controller: service.ScopedList(c)  ← 开发者写 1 行
      │
      │  内部自动执行：
      │  ├── BuildDataScope → 决定能看哪些行
      │  ├── ApplyDataScope → 拼 WHERE 条件
      │  ├── Count + Paginate → 分页查询
      │  ├── SanitizeResponse → 被 deny 的字段反射清零
      │  └── ApiSuccess → 返回标准响应
      │
      ▼
  ⑤ 返回 JSON 给前端（已脱敏、已分页）
```

## 4. 实战：一步步添加"采购中心"功能

### 业务场景

新增一个"采购中心"模块。部署后管理员在「角色与权限」页面能看到它、配置它。

---

### 步骤 1：定义数据模型

📁 **新建文件** `model/procurement_order.go`

```go
package model

type ProcurementOrder struct {
    Id           int     `json:"id" gorm:"primaryKey;comment:主键ID"`
    AgentId      int     `json:"agent_id" gorm:"index;comment:代理商ID"`
    UserId       int     `json:"user_id" gorm:"index;comment:下单人"`
    Sku          string  `json:"sku" gorm:"type:varchar(64);comment:商品型号"`
    Qty          int     `json:"qty" gorm:"comment:数量"`
    AgentPrice   float64 `json:"agent_price" gorm:"type:decimal(18,6);comment:代理商售价"`
    SupplierCost float64 `json:"supplier_cost" gorm:"type:decimal(18,6);comment:供应商成本"`
    Status       string  `json:"status" gorm:"type:varchar(32);comment:订单状态"`
    ApprovedBy   int     `json:"approved_by" gorm:"comment:审批人"`
    CreatedAt    int64   `json:"created_at" gorm:"autoCreateTime:milli;comment:创建时间"`
}

func (ProcurementOrder) TableName() string { return "procurement_orders" }
```

> `gorm:"comment:xxx"` 的中文注释会作为管理后台「字段权限」界面的字段显示名。

📁 **编辑** `model/main.go`，在 `AutoMigrate` 里追加 `&ProcurementOrder{}`。

---

### 步骤 2：定义 Feature

📁 **编辑** `model/permission_seed.go`，在 `defaultFeatures` 末尾追加：

```go
{
    FeatureKey:  "procurement",
    FeatureName: "采购中心",
    Module:      "platform",
    Namespace:   "platform",
    RoutePath:   "/console/procurement",
    Icon:        "Truck",
    Actions:     []string{"menu", "view", "create", "update", "delete", "approve", "export"},
    Resources:   []string{"procurement_orders"},
    UIElements: []uiElementSeed{
        {Key: "create",  Name: "新建订单", Type: "button"},
        {Key: "approve", Name: "审批",     Type: "button"},
        {Key: "delete",  Name: "删除",     Type: "button"},
        {Key: "export",  Name: "导出",     Type: "button"},
    },
},
```

> **注意**：`FeatureSeed` 没有 `IsSystem` 字段。系统内置功能的 `IsSystem` 标记由 `EnsureDefaultFeatures()` 在写入数据库时自动设置为 `true`。
>
> **不要加到 `rolePermMapping`**——我们要"默认不启用，管理员手动配置"。

### 步骤 5：配置路由（声明资源绑定 + 权限码）

📁 **编辑** `router/api-router.go`，追加路由组：

```go
procurement := apiRouter.Group("/procurement")
procurement.Use(
    middleware.UserAuth(),
    middleware.WithResource("procurement_orders"),  // ← 一行绑定，五道锁自动生效
)
{
    procurement.GET("/orders",
        middleware.RequirePerm(constant.PermPlatformProcurementView),
        controller.GetProcurementOrders)

    procurement.POST("/orders",
        middleware.RequirePerm(constant.PermPlatformProcurementCreate),
        controller.CreateProcurementOrder)

    procurement.PUT("/orders/:id",
        middleware.RequirePerm(constant.PermPlatformProcurementUpdate),
        controller.UpdateProcurementOrder)

    procurement.POST("/orders/:id/approve",
        middleware.RequirePerm(constant.PermPlatformProcurementApprove),
        controller.ApproveProcurementOrder)

    procurement.DELETE("/orders/:id",
        middleware.RequirePerm(constant.PermPlatformProcurementDelete),
        controller.DeleteProcurementOrder)

    procurement.GET("/orders/export",
        middleware.RequirePerm(constant.PermPlatformProcurementExport),
        controller.ExportProcurementOrders)
}
```

**WithResource Options（特殊表才需要）**：

```go
// 有自定义 scope 函数的表（如 logs 用 ApplyLogDataScope）
middleware.WithResource("logs", middleware.WithScopeFunc(model.ApplyLogDataScope))

// 没有 agent_id 列的表（如 tasks 通过子查询关联）
middleware.WithResource("tasks",
    middleware.WithScopeFunc(model.ApplyTaskDataScope),
    middleware.WithAgentColumn(""),
)

// 自定义默认排序
middleware.WithResource("channels", middleware.WithOrder("priority desc, id desc"))

// 跳过字段脱敏（如某些资源不需要字段级权限控制）
middleware.WithResource("xxx", middleware.WithoutSanitize())
```

| Option | 作用 | 默认值 |
|---|---|---|
| `WithScopeFunc(fn)` | 自定义数据范围 SQL 拼接 | `model.ApplyDataScope`（按 agent_id 过滤） |
| `WithAgentColumn(col)` | 指定 agent_id 列名 | `"agent_id"` |
| `WithOrder(order)` | 自定义默认排序 | `"id desc"` |
| `WithoutSanitize()` | 跳过字段脱敏 | `false`（默认脱敏） |

### 步骤 6：编写 Controller（这是最爽的部分）

📁 **新建文件** `controller/procurement.go`

```go
package controller

import (
    "github.com/QuantumNous/new-api/model"
    "github.com/QuantumNous/new-api/service"
    "github.com/QuantumNous/new-api/common"
    "github.com/gin-gonic/gin"
    "gorm.io/gorm"
)

// GET /api/procurement/orders — 列表（1 行搞定）
func GetProcurementOrders(c *gin.Context) {
    service.ScopedList[model.ProcurementOrder](c)
}

// GET /api/procurement/orders — 带搜索（3 行过滤）
func SearchProcurementOrders(c *gin.Context) {
    keyword := c.Query("keyword")
    service.ScopedList[model.ProcurementOrder](c, func(db *gorm.DB) *gorm.DB {
        if keyword != "" {
            return db.Where("sku LIKE ?", keyword+"%")
        }
        return db
    })
}

// GET /api/procurement/orders/:id — 详情（1 行搞定）
func GetProcurementOrder(c *gin.Context) {
    service.ScopedGet[model.ProcurementOrder](c)
}

// POST /api/procurement/orders — 创建
func CreateProcurementOrder(c *gin.Context) {
    var req model.ProcurementOrder
    if err := common.DecodeJson(c.Request.Body, &req); err != nil {
        common.ApiErrorMsg(c, "参数错误")
        return
    }
    req.UserId = service.MustUserId(c)
    req.AgentId = service.MustAgentId(c)
    req.Status = "draft"
    if err := model.DB.Create(&req).Error; err != nil {
        common.ApiError(c, err)
        return
    }
    common.ApiSuccess(c, req)
}

// POST /api/procurement/orders/:id/approve — 审批
func ApproveProcurementOrder(c *gin.Context) {
    id := c.Param("id")
    result := service.ScopedDB(c).
        Model(&model.ProcurementOrder{}).
        Where("id = ?", id).
        Updates(map[string]interface{}{
            "status":      "approved",
            "approved_by": service.MustUserId(c),
        })
    if result.RowsAffected == 0 {
        common.ApiErrorMsg(c, "订单不存在或无权操作")
        return
    }
    common.ApiSuccess(c, gin.H{"id": id})
}

// DELETE /api/procurement/orders/:id — 删除
func DeleteProcurementOrder(c *gin.Context) {
    id := c.Param("id")
    if err := service.ScopedDB(c).Delete(&model.ProcurementOrder{}, id).Error; err != nil {
        common.ApiError(c, err)
        return
    }
    common.ApiSuccess(c, nil)
}

// GET /api/procurement/orders/export — 导出
func ExportProcurementOrders(c *gin.Context) {
    items, err := service.ScopedExport[model.ProcurementOrder](c)
    if err != nil {
        common.ApiError(c, err)
        return
    }
    writeCSV(c, items) // items 已脱敏
}
```

### 框架 API 速查

| 函数 | 用途 | 行级 | 列级 | Controller 写几行 |
|---|---|---|---|---|
| `ScopedList[T](c)` | 分页列表 | 自动 | 自动 | **1 行** |
| `ScopedList[T](c, filterFn...)` | 带搜索的列表 | 自动 | 自动 | **3 行** |
| `ScopedListFn[T](c, queryFn)` | 复杂列表（JOIN/子查询） | 自动 | 自动 | **~10 行** |
| `ScopedGet[T](c)` | 单条详情（反射 AgentId） | 自动 | 自动 | **1 行** |
| `ScopedGetFn[T](c, getterFn, agentExtractor)` | 自定义详情 | 手动 | 自动 | **3 行** |
| `ScopedFetch[T](c, getterFn, agentExtractor)` | 写接口取记录（URL :id） | 手动 | 无 | **3 行** |
| `ScopedFetchRaw[T](c, id, getterFn, agentExtractor)` | 写接口取记录（body id） | 手动 | 无 | **3 行** |
| `ScopedDB(c)` | 获取 scoped DB | 自动 | 手动 | **1 行** |
| `ScopedDBFor(c, resource, applyFn)` | 不依赖中间件的 scoped DB | 自动 | 手动 | **1 行** |
| `ScopedExport[T](c, filterFn...)` | 导出（不分页） | 自动 | 自动 | **1 行** |
| `ScopedExportFn[T](c, queryFn)` | 自定义导出 | 自动 | 自动 | **~5 行** |
| `VerifyRecordScope(c, agentId)` | 单条记录归属校验 | — | — | **1 行** |
| `ScopeSnapshot(c)` | 异步任务快照 | 快照 | 快照 | **1 行** |
| `SanitizeAndRespond[T](c, item)` | 单条脱敏 + ApiSuccess | — | 自动 | **1 行** |
| `SanitizeResponse(c, resource, items)` | 手动列级脱敏（批量） | — | 手动 | **1 行** |
| `MustAgentId(c)` / `MustUserId(c)` | 获取上下文 ID | — | — | **1 行** |

### 选型决策树：我该用哪个函数？

```
你要写的是什么接口？
│
├── 读列表（分页）
│   ├── 查询是简单 WHERE？
│   │   └── YES → ScopedList[T](c, filterFn...)           ★ 首选
│   │
│   └── 查询有 JOIN / 子查询 / 非标准 count？
│       └── YES → ScopedListFn[T](c, queryFn)             需要自己写查询回调
│
├── 读单条
│   ├── model 有 AgentId 字段 + 用主键查就够？
│   │   └── YES → ScopedGet[T](c)                         ★ 首选
│   │
│   └── 需要 Preload / JOIN / 非标准查询？
│       └── YES → ScopedGetFn[T](c, getterFn, agentExtractor)
│
├── 写操作（Update / Delete / Manage）
│   ├── ID 从 URL :id 取？
│   │   └── YES → ScopedFetch[T](c, getterFn, agentExtractor)
│   │            拿到记录后自己处理业务逻辑
│   │
│   └── ID 从 request body 取？
│       └── YES → ScopedFetchRaw[T](c, id, getterFn, agentExtractor)
│
├── 写操作（Create）
│   └── 不需要 scope 查询，用 MustAgentId(c) 写入 agent_id
│       创建后需要脱敏返回？ → SanitizeAndRespond[T](c, &item)
│
├── 导出
│   ├── 简单全表导出？ → ScopedExport[T](c)
│   └── 复杂查询？    → ScopedExportFn[T](c, queryFn)
│
├── 异步导出任务
│   └── ScopeSnapshot(c) 拍快照，后台 goroutine 用快照执行
│
├── 需要 scoped DB 但逻辑太特殊（聚合/批量更新）
│   ├── 路由有 WithResource？ → ScopedDB(c)
│   └── 路由没有 WithResource？ → ScopedDBFor(c, "resource", applyFn)
│
└── 手动脱敏（老接口 / 不走框架的场景）
    └── SanitizeResponse(c, "resource", items)
```

**agentExtractor 参数说明**：

```go
// 告诉框架如何从记录中提取 agent_id，用于归属校验
func(u *model.User) int { return u.AgentId }

// 如果表没有 agent_id（如平台级资源），传 nil 跳过归属校验
service.ScopedGetFn(c, getterFn, nil)
```

**ScopedFetch vs ScopedGetFn 的区别**：

```
ScopedGetFn  = fetch + verify + sanitize + respond  （读接口，一步到位返回响应）
ScopedFetch  = fetch + verify                        （写接口，拿到记录后自己处理）
```

### 步骤 7-10：前端

前端部分不受框架改造影响，仍然使用：

- **菜单**：侧边栏配置加 `menuCode: 'menu:procurement'`，`hasMenuPermission()` 自动过滤
- **API 权限守卫**：`<PermissionGate code="platform:procurement:create">` — 没权限时渲染 fallback
- **UI 元素守卫**：`<UIGate feature="procurement" element="create">` — 基于 UI 权限控制按钮/筛选器
- **Tab 守卫**：`<GatedTabPane feature="procurement" element="tab_xxx">` — 不破坏 Semi UI 的 Tab 子元素检测
- **Hook 形式**：`const canExport = useUIPermission("procurement", "export")` — 在非 JSX 逻辑中判断
- **表格列**：`useFieldPermColumns(allColumns, 'procurement_orders')` 自动隐藏 deny 列（**必须接入**，见下方规则）
- **列选择器**：`getDeniedFields('procurement_orders')` 过滤不可见列
- **字段权限码**：`<PermissionGate code="field:procurement_orders:supplier_cost">` — 支持 `field:resource:field` 格式

**前端权限刷新**：保存角色配置后调 `permissionStore.refresh()` 重新拉取，或提示用户重新登录。

### 前端表格字段权限接入规则（必读）

#### 为什么必须接入

后端 `SanitizeResponse` 将 denied 字段**清零**（数字→0，字符串→空，时间→epoch），而不是从 JSON 中删除字段。如果前端不用 `useFieldPermColumns` 隐藏对应列，用户会看到 `¥0.00`、`1970-01-01 08:00:00`、`-` 等零值，而非列被隐藏。

```
后端返回:  { "money": 0, "create_time": 0, "trade_no": "ref_xxx" }
                   ↑ 被脱敏为 0           ↑ epoch 时间

前端不接入:  | 订单号       | 支付金额 | 创建时间           |
             | ref_xxx     | ¥0.00   | 1970-01-01 08:00  |  ← 用户困惑

前端接入后:  | 订单号       |                                   |
             | ref_xxx     |                                   |  ← 列直接隐藏
```

#### 接入方法

```jsx
import { useFieldPermColumns } from '../../hooks/common/useFieldPermColumns';

// 在 columns 定义之后、Table 渲染之前：
const visibleColumns = useFieldPermColumns(columns, 'procurement_orders');
//                                                   ↑ 对应后端 AutoRegisterModel 的资源名

<Table columns={visibleColumns} dataSource={data} />
```

#### 判断规则：什么表格需要接入

| 条件 | 需要接入？ | 说明 |
|------|-----------|------|
| 从 `WithResource` 保护的 API 读取数据 | **是** | 后端会对返回数据做 SanitizeResponse |
| 从无 `WithResource` 的 API 读取数据 | 否 | 不会被脱敏 |
| 纯配置编辑器（定价编辑、设置页） | 否 | 编辑表单，不是只读列表 |
| 操作型弹窗（测试、批量操作） | 否 | 不展示敏感数据 |
| 系统管理（角色、功能配置） | 否 | 字段无敏感性 |

**简单判断法**：如果表格的数据来源 API 路由上有 `middleware.WithResource("xxx")`，前端就必须 `useFieldPermColumns(columns, 'xxx')`。

| 选项 | SQL 效果 | 优先级（高→低） |
|---|---|---|
| `all` | 不加 WHERE | 1（最高） |
| `custom` | `WHERE agent_id IN (手动指定)` | 2 |
| `agent` | `WHERE agent_id = 当前代理商` | 3 |
| `self_and_sub` | `WHERE user_id IN (自己, 下属...)` | 4 |

>     ScopeType string // all, agent, self, self_and_sub, custom
>     AgentId   int
>     UserId    int
>     UserIds   []int  // self_and_sub 时的用户列表
>     AgentIds  []int  // custom 时的代理商列表
> }
> ```

### 5.5 Tab 4：交互权限

勾选的 UI 元素才渲染，未勾选的按钮/筛选器不显示。

### 5.6 Tab 5：可分配角色

### 5.7 配置后的生效

保存后受影响的用户需重新登录（或前端调 `permissionStore.refresh()`）。后端 Casbin 权限缓存 30 秒 TTL，UI 权限缓存 30 秒 TTL。

### 5.8 Casbin 策略引擎

权限检查由 Casbin 执行（`service/casbin.go` + `casbin_enforcer.go`）：

- **模型**：RBAC with domain（agent_id 作为 domain 实现多租户隔离）
- **Matcher**：`g(r.sub, p.sub, r.dom) && r.dom == p.dom && (r.obj == p.obj || p.obj == "*") && (r.act == p.act || p.act == "*")`
- **通配符**：`platform:*` 匹配所有权限，`*` 动作匹配所有操作
- **传播**：系统角色（agent_id=0）的策略自动传播到所有租户 domain

> **注意**：权限码格式 `{namespace}:{feature}:{action}` 在 Casbin 中拆分为 `obj = {namespace}:{feature}` + `act = {action}`。

## 7. 常见坑位与调试技巧

### 7.1 "部署了但管理后台看不到新功能"

检查 `permission_seed.go` 里有没有加 feature 定义 → 检查数据库 `permission_features` 表。

### 7.2 "改了权限但用户端没变化"

用户需重新登录。后端缓存 30 秒 TTL。检查 `localStorage.user` 里的权限。

### 7.3 "查询返回空列表"

数据范围未配置 → `WHERE 1=0`。去管理后台配置数据范围。

### 7.4 "字段全是零值"

字段权限白名单，新资源没勾选字段 → 全部 deny。去管理后台勾选。

### 7.5 "表格显示 ¥0.00、1970-01-01 等零值"

**这是最常见的遗漏**。后端 `SanitizeResponse` 将 denied 字段清零（不是删除），前端如果没用 `useFieldPermColumns` 隐藏对应列，零值就会原样渲染。

修复：在 `<Table>` 渲染前加 `useFieldPermColumns(columns, 'resource_name')`。详见上方「前端表格字段权限接入规则」。

### 7.6 "用 root 账号测试看不到脱敏效果"

root 持有 `platform:*`，跳过脱敏。用非 root 账号测试。

### 7.7 "RequireAnyPerm 和 RequirePerm 的区别"

- `RequirePerm(code)`：要求精确匹配一个权限码
- `RequireAnyPerm(codes...)`：多个权限码满足任意一个即可通过

### 7.8 "JSON blob 字段（如 Other）内的子字段无法控制"

需要在 `AutoRegisterModel` 时用 `WithJsonBlobField` + `WithJsonKeys` 显式注册子字段。

### 7.9 调试技巧

**后端**：
```go
// 打印当前用户的全部权限码
codes, _ := service.GetUserPermissions(userId, agentId)
log.Printf("codes=%v", codes)

// 打印当前用户的分类权限（菜单/API/角色等）
permsByType, _ := service.GetUserPermissionsByType(userId, agentId)
log.Printf("permissions=%+v", permsByType)

// 检查单个权限
has, _ := service.HasPermission(userId, agentId, "platform:procurement:view")
log.Printf("has=%v", has)

// 打印 SQL（调试数据范围）
service.ScopedDB(c).Debug().Find(&items)
```

**前端**：

|---|---|---|
| 查询涉及 JOIN/子查询 | `ScopedListFn` | token 列表 JOIN users 取 username |
| 返回 DTO 而非 model struct | `ScopedListFn` 泛型传 DTO 类型 | `ScopedListFn[model.TokenWithUser]` |
| 跨资源聚合（当前路由绑定的不是目标表） | `ScopedDBFor(c, "resource", applyFn)` | quota_data 用 logs 的权限配置但不同的 scope 函数 |
| 表没有 agent_id，通过 user_id 子查询关联 | `WithScopeFunc(ApplyTaskDataScope)` + `WithAgentColumn("")` | tasks、midjourney |
| 老接口不走 WithResource 中间件 | `ScopedDBFor` 或手动 `BuildDataScope` + `SanitizeResponse` | 兼容期过渡 |
| 创建/更新后返回单条脱敏数据 | `SanitizeAndRespond[T](c, &item)` | 创建后返回脱敏后的记录 |
| 第三方返回的 map | 先转 struct 再 `SanitizeResponse` | — |
| 写接口的字段黑名单 | Controller 里硬编码（本框架不管写入侧） | — |
| JSON blob 内的子字段权限控制 | `AutoRegisterModel` + `WithJsonBlobField` + `WithJsonKeys` | logs 的 Other 字段内 admin_info |

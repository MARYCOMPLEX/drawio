ROLES 四大系统职责划分
Gin
接入层 — 负责"进"和"出"
  • 接收 API 请求
  • 鉴权，识别代理商/终端用户/模型
  • 调 Billing 服务做预扣/实扣
  • 转发到上游模型
  • 返回结果给用户
  • 提供报表查询接口（查 ClickHouse）
PostgreSQL
交易真账系统 — 负责"真账"
  • 账户余额（可用/冻结）
  • 预扣 PREAUTH
  • 实扣 SETTLE
  • 退款 REFUND / 补扣 EXTRA_CHARGE
  • 账本流水 ledger_entry
  • 最终结算 request_settlement
Kafka
事件总线 — 负责"送消息"
  • 交易系统和统计系统解耦
  • 不让报表查询影响扣费链路
  • 支持异步消费、补消费、重放
  • Outbox 模式保证不丢事件
ClickHouse
分析库 — 负责"统计"
  • 存消费明细 fact_request_settlement
  • 分钟聚合 agg_minute
  • 日聚合 agg_daily
  • 代理商利润查询
  • 平台利润查询
  • 模型维度收益分析
FORMULAS 核心业务公式
用户侧
用户实扣 = f(代理商售价规则, 实际token)
终端用户最终被扣金额
代理商侧
代理商结算 = f(平台结算规则, 实际token)
代理商向总平台支付金额
平台侧
平台真实成本 = 上游真实计费金额
实际调用供应商花费
PROFIT 利润公式
代理商利润
= 用户实扣 - 代理商结算
代理商每笔请求的差价收入
平台利润
= 代理商结算 - 平台真实成本
平台每笔请求的差价收入
总毛利
= 用户实扣 - 平台真实成本
= 代理商利润 + 平台利润
TRUTHS 系统里的三份"真相"
第一份：请求真相
request_order
谁发起的？用了什么模型？请求状态如何？
回答"请求本身发生了什么"。
第二份：账务真相
ledger_entry + account_balance
谁被预扣？谁被实扣？谁被退款？余额剩多少？
回答"钱到底怎么扣的"。
第三份：统计真相
request_settlement + ClickHouse
代理商赚多少？平台赚多少？日报月报怎么出？
回答"对利润统计贡献了什么"。
